# GKE Kubernetes provisioner backend

> Provision Nora agents into Google Kubernetes Engine using Nora's generic Kubernetes adapter and the GKE Compose overlay.

GKE uses the same `k8s` backend id as every Kubernetes provider. Nora ships a single Kubernetes adapter (`workers/provisioner/backends/k8s.ts`) that serves Kind, K3s, AKS, GKE, and EKS — provider-specific overlays differ only in the kubeconfig they mount and the exposure mode they default to. The GKE overlay mounts a GKE kubeconfig and defaults agent Services to cloud `LoadBalancer` exposure.

## Step-by-step setup

### 1. Prerequisites

- GCP project with the Kubernetes Engine API enabled.
- `gcloud` CLI installed and `gcloud auth login` completed.
- An existing GKE cluster (or create one — GCP Console → **Kubernetes Engine → Clusters → Create**).
- `kubectl` installed on the host that runs Nora.

<Frame caption="GCP Console — Kubernetes Engine cluster details">
  ![GKE cluster details](/images/provisioner-backends/k8s/gke/cluster-details.png)
</Frame>

### 2. Create the kubeconfig

```bash theme={null}
mkdir -p .secrets
KUBECONFIG=./.secrets/gke-kubeconfig \
  gcloud container clusters get-credentials <cluster-name> \
    --region <region> \
    --project <project-id>
```

Use `--zone <zone>` instead of `--region <region>` for zonal clusters. You can copy the exact command from the cluster's **Connect** dialog.

<Frame caption="GCP Console — Cluster → Connect dialog (get-credentials command)">
  ![GKE connect dialog](/images/provisioner-backends/k8s/gke/connect-dialog.png)
</Frame>

### 3. Verify access from the host

```bash theme={null}
kubectl --kubeconfig ./.secrets/gke-kubeconfig get nodes
```

### 4. Configure Nora

Set the GKE helper variables in `.env`:

```bash theme={null}
GKE_KUBECONFIG_PATH=./.secrets/gke-kubeconfig
GKE_ENABLED_BACKENDS=k8s
GKE_K8S_NAMESPACE=openclaw-agents
GKE_K8S_EXPOSURE_MODE=load-balancer
GKE_K8S_LOAD_BALANCER_SOURCE_RANGES=<nora-control-plane-egress-cidr>
GKE_K8S_LOAD_BALANCER_READY_TIMEOUT_MS=1200000
GKE_DEPLOYMENT_JOB_TIMEOUT_MS=1800000
GKE_PROVISION_TIMEOUT_MS=1740000
```

Confirm region / VPC alignment in the Console's **Networking** tab before exposing services to the public internet.

<Frame caption="GCP Console — Cluster → Networking (region + VPC)">
  ![GKE networking](/images/provisioner-backends/k8s/gke/networking.png)
</Frame>

### 5. Start the stack

```bash theme={null}
docker compose -f docker-compose.yml -f docker-compose.gke.yml up -d --build
```

### 6. Deploy a test agent

Open the dashboard at `http://127.0.0.1:8080`, sign in, and create an agent with **Backend = Kubernetes**.

## GKE Service options

The default GKE `LoadBalancer` Service path works without annotations. To request GKE's external passthrough Network Load Balancer with RBS on supported cluster versions, set:

```bash theme={null}
GKE_K8S_LOAD_BALANCER_CLASS=networking.gke.io/l4-regional-external
```

For clusters that use the annotation path instead of `loadBalancerClass`:

```bash theme={null}
GKE_K8S_SERVICE_ANNOTATIONS_JSON='{"cloud.google.com/l4-rbs":"enabled"}'
```

Leave `GKE_K8S_SERVICE_ANNOTATIONS_JSON` empty unless your cluster policy requires provider-specific Service annotations.

## Automated smoke

Once your GKE kubeconfig is in place, run the shared lifecycle smoke:

```bash theme={null}
cd e2e
KUBECONFIG_PATH=$PWD/../.secrets/gke-kubeconfig npm run smoke:k8s-gke
```

This script is operator-run only — it provisions real workloads against a live GKE cluster and is not part of CI. Set `KEEP_ENV=true` to leave the stack running after the script finishes.

## Promote to production

Once the smoke is green, switch from the dev-mode stack to the prod-mode stack. The only changes vs the testing setup are nginx (public config + TLS) and Compose mode (auto-loaded `docker-compose.override.yml` for prod Dockerfiles + restart policies).

### 1. Tighten the LoadBalancer source ranges

Replace the smoke-time `<nora-control-plane-egress-cidr>` placeholder with your production control plane's egress CIDR. Combine with `cloud.google.com/load-balancer-type: Internal` annotations if you're keeping agents on a private subnet.

### 2. Switch nginx to public + TLS

In `.env`:

```bash theme={null}
NGINX_CONFIG_FILE=nginx.public.conf
NGINX_HTTP_PORT=80
```

Provision Let's Encrypt certs once:

```bash theme={null}
sudo ./infra/setup-tls.sh <your-domain>
```

### 3. Stop the smoke-mode stack

```bash theme={null}
docker compose -f docker-compose.yml -f docker-compose.gke.yml down
```

### 4. Start the prod-mode stack

```bash theme={null}
docker compose \
  -f docker-compose.yml \
  -f docker-compose.override.yml \
  -f docker-compose.gke.yml \
  up -d --build
```

### 5. Confirm

```bash theme={null}
docker compose ps
curl -fsS https://<your-domain>/api/health
docker compose logs worker-provisioner | tail -50
```

Deploy a real agent from the UI and confirm it reaches `running` and the GKE LoadBalancer assigns an external IP.
